# API Contract Delta: Tasks CRUD and Activity Log (003)

Extends [001's api.md](../../001-sheets-schema-and-api/contracts/api.md). Transport,
envelope, error codes, and auth (002) are **unchanged**: `text/plain` POST of
`{ token, action, payload }`, always HTTP 200, `ok` discriminates, every non-`ping` action
requires a verified allowlisted token. This document adds three actions, tightens two, and
supersedes one line of 001's task semantics.

## New & changed actions

| Action | Payload | Data returned | Change |
|--------|---------|---------------|--------|
| `tasks.list` | `{ filter? }` — `mine\|theirs\|ours\|all\|default` (default `all`) | `{ tasks: [Task…] }` | **tightened**: adds server-side identity-relative filtering |
| `tasks.update` | `id` + any of `title\|owner\|dueDate` | `{ task }` | **tightened**: `status`/`completedBy`/`completedAt` now rejected |
| `tasks.complete` | `{ id }` | `{ task, changed }` | **new** |
| `tasks.reopen` | `{ id }` | `{ task, changed }` | **new** |
| `activity.list` | `{ limit?, since? }` | `{ activity: [FeedEntry…] }` | **new** |

`tasks.create` and `tasks.delete` are unchanged from 001 (create still defaults `status` to
`open` and ignores any client `completedBy`/`completedAt`; delete is still a hard delete
whose ActivityLog `detail` preserves the title).

## Semantics

### `tasks.list` — slices (FR-008/009/010)

- `filter` resolves **server-side against the verified caller** — never a client "who am I"
  (FR-009). Personal account → its `max`/`jaz` identity. Shared account → `payload.actingPerson`
  (`max`/`jaz`), **required** for `mine`/`theirs`/`default`, else `ACTING_PERSON_REQUIRED`;
  not required for `all`/`ours`.
- Membership: `mine` = owner is exactly the caller; `theirs` = owner is exactly the other
  person; `ours` = owner is `both`; `all` = every task; `default` = `mine ∪ ours`. `both`
  tasks appear in `ours`/`default` only — the three named slices are disjoint and union to
  `all`.
- Slices include tasks of **all** statuses (owner-only filtering); no status/date filtering
  in this feature.
- Unknown `filter` value → `VALIDATION_FAILED` (field `filter`). Omitted `filter` → `all`
  (preserves 001 behavior).

### `tasks.complete` (FR-002/003)

- On an `open` or `snoozed` task: set `status = done`, stamp `completedBy` = verified actor,
  `completedAt` = now (household tz); append one ActivityLog row with **`action: "complete"`**.
  Returns `{ task, changed: true }`.
- On an already-`done` task: **no-change** — original `completedBy`/`completedAt` preserved,
  **no** new log row. Returns `{ task, changed: false }` with the unchanged task.
- A single completion closes a `both`-owned task.
- Read-then-write happens inside the write lock, so the simultaneous-completion race resolves
  to exactly one completion + one feed entry (the loser gets `changed: false`).
- Unknown `id` → `NOT_FOUND`.

### `tasks.reopen` (FR-004)

- On a `done` task: set `status = open`, clear `completedBy`/`completedAt`; append one
  ActivityLog row with **`action: "reopen"`**. Returns `{ task, changed: true }`. Either user
  may reopen any task regardless of who completed it. The prior completion remains in the log.
- On an already-`open`/`snoozed` task: no-change, no log row, `{ task, changed: false }`.
- Unknown `id` → `NOT_FOUND`.

### `tasks.update` — tightened (FR-005, FR-015)

- Mutable fields: `title`, `owner`, `dueDate` only. Partial (only supplied fields change);
  `dueDate` may be set, moved, or **cleared** (empty string clears it).
- Supplying `status`, `completedBy`, or `completedAt` → `BAD_REQUEST` naming the dedicated
  action ("use tasks.complete / tasks.reopen"). This **supersedes** 001's api.md line
  *"Setting `status:"done"` stamps completedBy … back to `open` clears both"* — completion has
  exactly one path now.
- Logs `action: "update"` (unchanged), so completions never masquerade as edits in the feed.

### `activity.list` — feed (FR-011/012/013/014)

- Read-only; requires a verified allowlisted caller; no `actingPerson`, no write lock.
- Returns ActivityLog **newest-first** (reverse append order), filtered by optional `since`
  (keep entries with `timestamp >= since`) and truncated to the most recent `limit`
  (**default 200**, hard max 500; a larger `limit` is clamped). More requested than exists →
  what exists. Empty log → `{ activity: [] }`.
- Each `FeedEntry`:

  ```json
  {
    "timestamp": "2026-07-08T14:14",
    "actor": "jaz",
    "action": "complete",
    "targetId": "…uuid…",
    "detail": "Buy flea meds",
    "summary": "Jaz completed 'Buy flea meds'"
  }
  ```

  `summary` is composed at read time (name map `max→Max`, `jaz→Jaz`, `system→System`,
  unknown actor passed through; verb map `create→added`, `update→edited`,
  `complete→completed`, `reopen→reopened`, `delete→deleted`, `adopt-id→assigned an id to`,
  `provision→set up`; quoted `detail` omitted when empty). Entries render regardless of
  whether `targetId` still exists (FR-013) and `system` entries read as System (FR-011).
- No action edits or deletes log rows (FR-014).

## Error codes

No new codes. Reuses 001/002's closed set: `VALIDATION_FAILED` (bad `filter`),
`BAD_REQUEST` (lifecycle field on `tasks.update`), `NOT_FOUND` (complete/reopen unknown id),
`ACTING_PERSON_REQUIRED` (shared-account identity-relative slice without `actingPerson`),
`BUSY`, `SCHEMA_MISMATCH`, `INTERNAL`.
