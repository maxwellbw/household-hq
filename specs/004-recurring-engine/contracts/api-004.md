# API Contract Delta: Recurring Chore Engine (004)

Extends [001's api.md](../../001-sheets-schema-and-api/contracts/api.md) and the
[003 delta](../../003-tasks-crud-and-activity-log/contracts/api-003.md). Transport,
envelope, error codes, and auth (002) are **unchanged**: `text/plain` POST of
`{ token, action, payload }`, always HTTP 200, `ok` discriminates, every non-`ping` action
requires a verified allowlisted token; shared-account **writes** need `actingPerson`
(`max`/`jaz`). This document adds three rule-management actions and specifies the nightly
generator (which is a trigger, not an API action).

## New actions

| Action | Payload | Data returned | Change |
|--------|---------|---------------|--------|
| `recurring.create` | `title`, `cadence`, `anchorDate`, `defaultOwner`, `seasonStart?`, `seasonEnd?`, `id?` | `{ recurring }` | **new** |
| `recurring.update` | `id` + any of `title\|cadence\|anchorDate\|defaultOwner\|seasonStart\|seasonEnd` | `{ recurring }` | **new** |
| `recurring.delete` | `{ id }` | `{ id }` | **new** |

`recurring.list` is unchanged from 001 (`{ recurring: [Rule…] }`, every rule, read-only
projection with `_warnings` on any unparseable cell).

## Field rules (all three write actions)

- **Allowed fields** are exactly the Recurring columns: `id, title, cadence, anchorDate,
  defaultOwner, lastGenerated, seasonStart, seasonEnd`. Any other key → `BAD_REQUEST`
  (`rejectUnknownFields_`).
- `cadence` ∈ `weekly|biweekly|monthly|quarterly|annually`, else `VALIDATION_FAILED`.
- `anchorDate` is `YYYY-MM-DD`, else `VALIDATION_FAILED`.
- `defaultOwner` ∈ `max|jaz|both`, else `VALIDATION_FAILED`.
- `seasonStart`/`seasonEnd` are integers 1–12 and **all-or-nothing** (both set or both
  blank); each 1–12; `start > end` is a legal wrap-around window; a half-set pair or an
  out-of-range month → `VALIDATION_FAILED` (`validateSeasonWindow_`).
- `lastGenerated` is **generator-managed**: supplying it on `recurring.create` or
  `recurring.update` → `BAD_REQUEST` (research D8). Clear it by hand-editing the Sheet to
  force re-generation.

## Semantics

### `recurring.create` (FR-001)

- Requires `title`, `cadence`, `anchorDate`, `defaultOwner`. Optional season pair.
- Client-supplied `id` makes the create idempotent (001 replay): a known id returns the
  existing rule, no duplicate. Blank/omitted `id` → generated UUID.
- `lastGenerated` starts blank, so the first generator run back-fills the watermark from
  `today` (no backlog of past occurrences).
- Appends one ActivityLog `create` row attributed to the acting user.
- Returns `{ recurring }` — the stored rule.

### `recurring.update` (FR-009)

- Requires `id`; patches only the provided editable fields (season pair validated together
  using the merged values so you can set/clear both at once).
- Does **not** touch already-generated Tasks (FR-008); only future occurrences reflect the
  change.
- Unknown `id` → `NOT_FOUND`. Appends one `update` row attributed to the acting user.
- Returns `{ recurring }` — the merged rule.

### `recurring.delete` (FR-010)

- Requires `id`; hard-deletes the rule row. Already-generated occurrence Tasks **remain** as
  ordinary tasks.
- Unknown `id` → `NOT_FOUND`. Appends one `delete` row (with the rule's title in `detail`)
  attributed to the acting user.
- Returns `{ id }`.

## Generator (trigger, not an API action) — `generateRecurringTasks_()`

Runs nightly (feature 007+ digests aside) and is also runnable from the editor. Not
reachable over HTTP. Per rule:

1. Read `recurringLookaheadDays` from Settings (default 30). Compute
   `windowStart = lastGenerated || today−1d`, `windowEnd = today + lookahead` (household tz).
2. Walk cadence occurrences from `anchorDate` within `(windowStart, windowEnd]`.
3. For each occurrence whose **month** is in-season (or the rule is year-round): create a
   Task with deterministic id `'r'+hex(MD5(recurringId+'|'+dueDate))`, `title`/`owner` from
   the rule, `status=open`, `recurringId` back-link. Reuses `createRecord_` →
   **idempotent** (re-run/overlap never duplicates, FR-006/007) and appends one `create` row
   by actor `system` (FR-011).
4. Advance `lastGenerated` to the newest occurrence **considered** in the window (created or
   season-skipped) via `updateRecordById_(…, 'system')` — logged, and only when it changes.
   Because the next run starts strictly after `lastGenerated`, a user-deleted occurrence is
   **never re-created** (FR-013).

Idempotency & concurrency: all writes go through `withLock_`; the deterministic id plus the
monotonic watermark mean any number of overlapping or repeated runs converge to exactly one
Task per occurrence (SC-002).

## Error codes

No new codes. Reuses `BAD_REQUEST`, `VALIDATION_FAILED` (with `field`), `NOT_FOUND`, `BUSY`,
`INTERNAL`, and the 002 auth codes (`UNAUTHENTICATED`, `FORBIDDEN`, `ACTING_PERSON_REQUIRED`,
`ALLOWLIST_MISCONFIGURED`).

## Versioning

Additive (new actions, one Settings row, no changed shapes) → `API_VERSION` minor bump
(`1.0.0` → `1.1.0`) at implementation, per 001's versioning note.
