# API Contract: Grocery & Household Lists (024)

Extends the base contract ([001's api.md](../../001-sheets-schema-and-api/contracts/api.md),
per subsequent features). Transport, envelope, error codes, and auth (002) are
**unchanged**: `text/plain` POST of `{ token, action, payload }`, always HTTP 200, `ok`
discriminates, every action requires a verified allowlisted token. This document adds two
new tabs (`Lists`, `ListItems` — see [data-model.md](../data-model.md)) and their actions.

## New actions

| Action | Payload | Data returned | Notes |
|---|---|---|---|
| `lists.list` | `{}` | `{ lists: List[] }` | All lists, unfiltered |
| `lists.create` | `{ name }` | `{ list: List }` | `name` required |
| `lists.delete` | `{ id }` | `{ id }` | Cascades: deletes every `ListItem` with matching `listId` first |
| `listItems.list` | `{ listId? }` | `{ items: ListItem[] }` | All items, or filtered to one list if `listId` given |
| `listItems.create` | `{ listId, name, section?, staple?, note? }` | `{ item: ListItem }` | See **Reuse-and-flip** below |
| `listItems.update` | `{ id, name?, section?, staple?, note? }` | `{ item: ListItem }` | `status` is **not** settable here — use `listItems.toggle` |
| `listItems.toggle` | `{ id }` | `{ item: ListItem, changed: boolean }` | Flips `need`⇄`stocked`; no other field changes |
| `listItems.delete` | `{ id }` | `{ id }` | Removes the row outright (not a status change) |

`List` shape: `{ id, name }`. `ListItem` shape:
`{ id, listId, name, status, section, staple, note }` (empty string for unset
`section`/`note`; `staple` always `'TRUE'`/`'FALSE'`).

## Semantics

### `lists.create` (FR-001/002)

`requireFields_(['name'])`, `rejectUnknownFields_`. No uniqueness check — duplicate list
names are legal (same convention as duplicate Event titles).

### `lists.delete` (FR-002, cascade)

`requireFields_(['id'])`. Fails `NOT_FOUND` if the list doesn't exist. On success, deletes
every `ListItems` row whose `listId` matches, then the `Lists` row itself — mirrors
`deleteEvent_`'s cascade to prep Tasks. Logs one `delete` row for the list; each cascaded
item delete logs its own row too (consistent with how every `deleteRecordById_` call
logs), so a list's removal is fully reconstructable from the feed.

### `listItems.create` — reuse-and-flip (FR-003/004/006/007, research R3)

1. `requireFields_(['listId', 'name'])`; `rejectUnknownFields_`.
2. Verify `listId` references an existing `Lists` row → `NOT_FOUND` if not.
3. Look up existing `ListItems` rows on that `listId` whose `name` matches the new
   `name`, trimmed and case-insensitive.
   - **Match found**: flip that row's `status` to `need` (no-op, no log row, if it's
     already `need`); ignore any `section`/`staple`/`note` supplied in this call (the
     existing row's values are untouched — edit via `listItems.update` if they need to
     change); return the existing (possibly updated) row.
   - **No match**: create a new row. `status` always starts `need` regardless of any
     client-supplied value (a non-`need` `status` in the payload is `BAD_REQUEST`, same
     pattern as `createTask_` rejecting a non-`open` create status). `section`/`staple`/
     `note` are optional and default to `''`/`FALSE`/`''`.
4. One ActivityLog row either way: `create` for a new row, `list-item-need` if an existing
   row was flipped, nothing if the existing row was already `need`.

### `listItems.update` (FR-008/009/010)

`requireFields_(['id'])`; `rejectUnknownFields_`. Any of `name`/`section`/`staple`/`note`
may be patched. A `status` key in the payload is rejected with `BAD_REQUEST` ("use
listItems.toggle"), mirroring how `tasks.update` rejects lifecycle fields. Logs one
`update` row (no-op if nothing actually changed, mirroring `updateRecordById_`'s existing
diff-before-write behavior elsewhere).

### `listItems.toggle` (FR-005, research R4)

`requireFields_(['id'])`. Flips `status`: `need`→`stocked` or `stocked`→`need`. No payload
beyond `id` — the server determines the new value from the current one. Idempotent
no-op-safe by construction (there is no "toggle to X" request to repeat). Logs one row per
actual flip: `list-item-need` or `list-item-stocked` (new `ACTION_VERBS` entries: "marked
needed" / "marked stocked"). Returns `{ item, changed }`; `changed: false` never happens
in practice (every call flips something) but the shape matches `tasks.complete`/`reopen`
for frontend consistency.

### `listItems.delete` (Edge Cases)

`requireFields_(['id'])`. Outright removal — distinct from toggling to `stocked`. Used for
"we don't buy that anymore," not for routine shopping cycles (FR-006 explicitly says
routine cycling never deletes).

## Settings

`groceryStapleNudgeThreshold` (default `3`) is added to `EDITABLE_SETTINGS` and
`SETTINGS_SEED`, validated as a positive integer (`posint`) in `validateSettingValue_`,
same pattern as `gcalEventReminderMin`. No new action — read via the existing
`settings.list`, written via the existing `settings.update`.

## Unchanged

The existing `Tasks.listItems` column (a delimited string, unrelated prior-feature
concept) is untouched by this feature — no action here reads or writes it.
