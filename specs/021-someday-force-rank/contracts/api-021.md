# API Contract Delta: Someday Force-Rank (021)

Extends [003's api-003.md](../../003-tasks-crud-and-activity-log/contracts/api-003.md). Transport,
envelope, error codes, and auth (002) are **unchanged**: `text/plain` POST of
`{ token, action, payload }`, always HTTP 200, `ok` discriminates, every action requires a
verified allowlisted token. This document adds **one** action and **one** column.

## Schema change

`Tasks` tab gains a trailing column **`somedayRank`** (see [data-model.md](../data-model.md)):
plain integer as string, **blank = unranked**. Existing rows/hand-added rows unaffected
(trailing column, blank tolerated). `Task` objects returned by every `tasks.*` action now
carry `somedayRank` (empty string when unset).

## New action

| Action | Payload | Data returned | Change |
|--------|---------|---------------|--------|
| `tasks.rank` | `{ order: string[] }` — Task IDs, highest priority first | `{ ranked: number }` — count written | **new** |

## Semantics

### `tasks.rank` (FR-009/FR-011/FR-016/FR-021/FR-022)

- **Payload**: `order` is an array of Task IDs in the desired shared priority order (index 0 =
  rank `1`). Missing/`non-array` `order` → `VALIDATION_FAILED` (field `order`). An **empty**
  array is legal and means "clear all someday ranks" (returns `{ ranked: 0 }`).
- **Write** (single batch, under `LockService`): read the Tasks tab once; for each ID in
  `order`, set `somedayRank` to its 1-based position (`String(index + 1)`); for any Task whose
  `somedayRank` is currently non-blank but is **not** in `order`, clear it to `''` (no phantom
  ranks — FR-021). Write the whole tab back in one batch.
- **Unknown IDs in `order`**: IDs not found in the Tasks tab are **skipped** (not an error) —
  the list may have drifted since the session started (concurrent schedule/complete/delete).
  Positions are assigned over the IDs that resolve, preserving their relative order; the returned
  `ranked` count reflects rows actually written.
- **No dueDate/status/owner side effects**: `tasks.rank` only touches `somedayRank`. It does
  **not** validate that the listed tasks currently qualify for Someday — it stores the order as
  given; qualification is a render-time concern (a scheduled task simply won't show in Someday
  despite a stored rank).
- **Idempotent**: submitting the same `order` twice yields the identical Sheet state and is safe
  to re-run (trigger-safe convention). Re-running produces a new ActivityLog row each time (an
  explicit user re-rank action), but no data drift.
- **ActivityLog**: append exactly **one** row — `action: "rank-someday"`, `actor` = verified
  caller, `targetId: "someday-rank"` (a stable household-level sentinel — there is no single
  target task), `detail` = number of tasks ranked (FR-022). Not one row per task.
- **Failure**: any write failure aborts the whole batch (lock released, nothing partially
  applied) and returns a non-`ok` envelope; the frontend surfaces "didn't save" and keeps the
  prior order on screen (FR-016). No partial order is ever presented as saved.

## Unchanged

`tasks.update` still accepts `title|owner|dueDate|notes` only in practice. Although
`somedayRank` is now a Tasks header (so `mutablePatch_` *could* carry it), the app never sends
`somedayRank` via `tasks.update` — `tasks.rank` is the canonical writer. `tasks.create` ignores
any client `somedayRank` (new tasks start unranked). Scheduling a someday task (`tasks.update`
with a `dueDate`) does not clear `somedayRank`; the stale value is inert until/unless the task
returns to Someday.
