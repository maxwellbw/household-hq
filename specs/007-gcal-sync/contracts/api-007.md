# Contract: api-007 — Calendar Sync Side Effects

Feature 007 adds **no new API actions**. It attaches an idempotent calendar-mirror side
effect to existing Events/Tasks write actions and adds one time-driven trigger entry point.
Transport, envelope, auth, and error codes are unchanged from
[api.md](../../001-sheets-schema-and-api/contracts/api.md) / 002 / 003 / 005.

## Affected existing actions (behavior added, response shape unchanged)

| Action            | Added side effect (best-effort, never fails the write) |
|-------------------|--------------------------------------------------------|
| `events.create`   | mirror the new event → Household calendar; store `gcalEventId` |
| `events.update`   | update the event's calendar entry in place |
| `events.delete`   | delete the event's calendar entry |
| `tasks.create`    | if `dueDate ≥ today`, mirror as an all-day entry; store `gcalEventId` |
| `tasks.update`    | re-mirror: move on date change, remove on date-clear, recolor on owner change |
| `tasks.complete`  | remove the task's calendar entry |
| `tasks.reopen`    | re-create the task's calendar entry |
| `tasks.delete`    | delete the task's calendar entry |

Notes:
- The mirror runs **after** the Sheet write succeeds, inside a `try/catch`. A calendar failure
  is logged (`console.error`) and swallowed — the API response is still `{ ok: true }` with the
  record. The nightly reconcile repairs any missed mirror (FR-010).
- Responses now include the populated `gcalEventId` on Events/Tasks once mirrored (it is an
  existing/near-existing header field; clients may ignore it).
- When `Settings.householdCalendarId` is blank, every side effect is a silent no-op (FR-014).

## New non-API entry points (Apps Script editor / trigger only — not in `HANDLERS`)

| Function                   | Kind | Purpose |
|----------------------------|------|---------|
| `syncCalendar()`           | time-driven trigger handler (public name, no `_`) | nightly full reconcile of all Events/Tasks + orphan sweep |
| `installCalendarTrigger()` | one-time editor run (public name, no `_`) | idempotently (re)install the single nightly trigger at `GCAL_TRIGGER_HOUR` |

Both are **public** (no trailing underscore) per CLAUDE.md — trigger handlers and editor-run
entry points must not use the private-underscore convention. Internal helpers
(`syncCalendarForEvent_`, `syncCalendarForTask_`, the locked cell-writer, builders) keep the
trailing underscore.

## Logging (Principle VI, FR-012)

Every calendar create/update/delete appends one ActivityLog row:

```
action:   gcal-sync
actor:    system
targetId: <event id | task id>
detail:   e.g. "created calendar event for 'Vet appointment'"
```

No-op mirrors (record already in sync) append nothing.

## Idempotency guarantees (Principle V; FR-009/015; SC-002/005)

- Re-running `syncCalendar()` with no app-side changes performs **zero** calendar writes.
- A given record maps to **exactly one** calendar entry (or zero) at all times — no duplicates
  across repeated edits/re-syncs.
- A stale pointer (calendar entry deleted in Google) is re-created on the next mirror, not
  crashed on.
- A partway-failed run reconverges on the next run with no duplicates and no lost updates.
