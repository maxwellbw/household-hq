# API Contract Delta: Events and Prep Templates (005)

Extends [001's api.md](../../001-sheets-schema-and-api/contracts/api.md), the
[003 delta](../../003-tasks-crud-and-activity-log/contracts/api-003.md), and the
[004 delta](../../004-recurring-engine/contracts/api-004.md). Transport, envelope, error
codes, and auth (002) are **unchanged**: `text/plain` POST of `{ token, action, payload }`,
always HTTP 200, `ok` discriminates, every non-`ping` action requires a verified allowlisted
token; shared-account **writes** need `actingPerson` (`max`/`jaz`). This document adds three
template-management actions, specifies prep side effects on the existing event actions, and
specifies the nightly prep generator (a trigger, not an API action).

## New actions

| Action | Payload | Data returned | Change |
|--------|---------|---------------|--------|
| `templates.create` | `eventType`, `taskTitle`, `offsetDays`, `defaultOwner`, `id?` | `{ template }` | **new** |
| `templates.update` | `id` + any of `eventType\|taskTitle\|offsetDays\|defaultOwner` | `{ template }` | **new** |
| `templates.delete` | `{ id }` | `{ id }` | **new** |

`templates.list` is unchanged from 001 (`{ templates: [Step…] }`, every step, read-only
projection with `_warnings` on any unparseable cell).

## Changed actions (Events — prep side effects added; shapes unchanged)

| Action | Payload | Data returned | Change in 005 |
|--------|---------|---------------|---------------|
| `events.create` | `title`, `start`, `end`, `owner`, `type?`, `templateId?`, `notes?`, `id?` | `{ event }` | Generates prep for `templateId` after create; sets `prepGeneratedFor`. |
| `events.update` | `id` + any of `title\|start\|end\|owner\|type\|templateId\|notes` | `{ event }` | Re-dates outstanding prep if `start` moved; swaps the prep set if `templateId` changed. |
| `events.delete` | `{ id }` | `{ id }` | **Purges all** prep tasks for the event (completed + outstanding). |

`events.list` unchanged. `events.create`/`events.update` now **reject** a client-supplied
`prepGeneratedFor` (below).

## Field rules

### Templates (`templates.*`)

- **Allowed fields** are exactly the TaskTemplates columns: `id, eventType, taskTitle,
  offsetDays, defaultOwner`. Any other key → `BAD_REQUEST` (`rejectUnknownFields_`).
- Required on create: `eventType`, `taskTitle`, `offsetDays`, `defaultOwner`.
- `offsetDays` is a signed integer (`int` type; e.g. `-2` = two days before), else
  `VALIDATION_FAILED`.
- `defaultOwner` ∈ `max|jaz|both`, else `VALIDATION_FAILED`.
- `eventType`/`taskTitle` are free text (any non-empty string).

### Events (`events.create` / `events.update`)

- Allowed fields are the Events columns **except** `prepGeneratedFor`, which is
  **generator-managed**: supplying it on create/update → `BAD_REQUEST` (research D9). Clear it
  by hand-editing the Sheet to force regeneration.
- `type` and `templateId` are free text. `templateId` need not match an existing `eventType`
  — a value with no matching steps is a legal empty checklist (generates no prep, not an error).
- Existing rules unchanged: `start`/`end` are `datetime`; `end >= start`; `owner` ∈
  `max|jaz|both`.

## Semantics

### `templates.create` / `templates.update` / `templates.delete` (US2 · FR-005/006/007)

- Standard row CRUD, mirroring 004's rule CRUD. Client-supplied `id` makes create idempotent
  (001 replay). Unknown `id` on update/delete → `NOT_FOUND`. Each appends one
  `create`/`update`/`delete` ActivityLog row attributed to the acting user.
- **No retroactive regeneration** (research D8): editing or deleting a step does not rewrite
  prep already generated for existing events; a new/edited step is applied the next time an
  event of that `eventType` is tagged, retagged, or generated.

### `events.create` (US1/US3 · FR-001/008)

- Creates the event (unchanged), then runs `syncPrepForEvent_` (D3): if `templateId` is set,
  materializes one prep task per matching `TaskTemplates` step and sets `prepGeneratedFor =
  templateId`. Each prep task: deterministic id `'p'+hex(MD5(eventId+'|'+stepId))`, `title`/
  `owner` from the step, `dueDate = dateOf(start)+offsetDays` (may be in the past — FR-018),
  `eventId` back-link, `status=open`. Prep creation is idempotent and logs one `create` per
  prep task by `system`.
- Returns `{ event }` — the stored event (with `prepGeneratedFor` set if prep ran).

### `events.update` (US1/US4 · FR-002/015/016)

- Patches the provided editable fields (unchanged), then runs `syncPrepForEvent_`:
  - **`start` moved** (template unchanged) → each outstanding prep task's `dueDate` is
    recomputed as `dateOf(newStart)+offsetDays`; completed prep is untouched (FR-015). A
    re-date logs one `update` by `system` per changed task (no change ⇒ no write, no log).
  - **`templateId` changed** (incl. cleared) → outstanding prep from the old checklist is
    removed, prep for the new checklist is generated, `prepGeneratedFor` advances; completed
    prep from the old checklist remains (FR-016).
- Returns `{ event }` — the merged event.

### `events.delete` (US4 · FR-017)

- Hard-deletes the event row **and** every prep task for it (`Tasks` where `eventId == id` and
  `id` matches `^p[0-9a-f]{32}$`), completed and outstanding alike. A user's manually
  event-linked tasks (non-prep ids) are **not** deleted. Each removal logs one `delete` by the
  acting user. Unknown `id` → `NOT_FOUND`.
- Returns `{ id }`.

## Generator (trigger, not an API action) — `generatePrepTasks()`

Runs nightly and is also runnable from the editor; not reachable over HTTP. For **every**
event it calls `syncPrepForEvent_(event, 'system')` (D3), which is idempotent:

1. An event whose `templateId != prepGeneratedFor` (a hand-added event, a hand-edited retag,
   or a synchronous run that never completed) → generate the desired prep set, retire the old
   template's outstanding prep, advance the marker. Deterministic ids + `createRecord_` mean
   re-runs/overlaps never duplicate (FR-011).
2. An event in steady state (`templateId == prepGeneratedFor`) → re-date outstanding survivors
   to the current start; **create nothing**, so a hand-deleted prep task is **not resurrected**
   (FR-014).

Idempotency & concurrency: all writes go through `withLock_`; the deterministic id plus the
`prepGeneratedFor` marker mean any number of overlapping or repeated runs converge to exactly
one prep task per event-and-step (SC-003). Installed once via `installPrepTrigger()` (editor
run; reuses 004's `script.scriptapp` scope — no new authorization).

## Error codes

No new codes. Reuses `BAD_REQUEST`, `VALIDATION_FAILED` (with `field`), `NOT_FOUND`, `BUSY`,
`INTERNAL`, and the 002 auth codes (`UNAUTHENTICATED`, `FORBIDDEN`, `ACTING_PERSON_REQUIRED`,
`ALLOWLIST_MISCONFIGURED`).

## Versioning

Additive (new actions, prep side effects on existing actions, one new column, no changed
response shapes) → `API_VERSION` minor bump (`1.1.0` → `1.2.0`) at implementation, per 001's
versioning note.
